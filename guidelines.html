Guidelines

Introduction

This guide will instruct you all the best practices which need to be followed during implementation of a feature, updating services, fixing bugs or new service creation.

  • Service to service communication

  • Distributed tracing

  • Distributed session

  • Logging

  • Session termination

  • Brute force protection

  • Private routing

  • Circuit breaker

Service to service communication

When communicating to other services withing nipige infra there should be at-least one or combination of these verification

  • tls

  • secrets

  • kms

Distributed tracing

All request should contain a request id header, which should be used to update request tracking data in one of these

  • OpenTelemetry

  • Jaeger

  • Zipkin

These requests should be checked periodically for continues failures and raise alerts etc on any kind of abnormality.

Recommendation: Create a library which can work as a request middleware handling, it will

  • automatically update tracing server upon receiving the request.

  • attach http client in request context which will attach required data in all outgoing requests.

Distributed sessions

Only entrypoint of the service need to validate the authentication and attach the role, tenant and user_id of caller to all outgoing request to another services, which can be used by other services to handle the authorization. This'll improve service response time along with reducing database load.

Logging

These kind of action should be always logged and system should have a trail of all the action perform by the user.

  • Login

  • Password reset

  • Payment

  • Deletions

  • Creations

  • Invalid requests

These are few examples, logging should be done to all kind of sensitive operations.

Use @nipige/logger for logging middleware, it is configured to make use of logger sidecar in pod which then records required data in influx db.

Session termination

Application should always have a way to allow user to all its logged in devices and logout one specifically if needed.

Brute force protection

These action should be restricted by number of failed attempts

  • Login

  • Forgot Password

  • Wrong OTP attempts

  • OTP requests (Should be one every 30 seconds only)

Private routing

  • Internal routes should not be accessible from outside the cluster.

  • Request withing the kubernetes services should be called directly using service name

  • For private request all base path of service should be /private/

  • Block ingress to proceed on request matching **/private/**/* path pattern

Circuit breaker

All services should follow circuit breaker pattern. It works by wrapping a potentially dangerous or faulty operation in a circuit breaker object. The circuit breaker is designed to detect when the operation is failing or taking too long to complete.

Once a threshold is reached, the circuit breaker will “trip” and stop the operation from executing, returning a pre-configured fallback value instead. This helps prevent further damage by stopping the faulty operation from cascading through the system.
